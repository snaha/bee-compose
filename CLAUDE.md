## What this repo is

A Docker Compose stack for running a local [Swarm](https://www.ethswarm.org/) Bee cluster (1 queen + up to 4 workers) against an Anvil dev chain that's pre-loaded with deployed Swarm contracts. There is **no application code** — the repo is `compose.yml`, two thin Dockerfiles (bee, blockchain), a baked Anvil state snapshot, pre-generated dev identities, and a handful of shell scripts.

## Common commands

```bash
# queen + chain (workers are behind the `workers` profile and stay off)
docker compose up -d

# add workers — resolves queen's peer id from /addresses and exports QUEEN_BOOTNODE
./scripts/workers-up.sh

# buy a postage stamp on the queen (defaults: amount 414720000, depth 20)
./scripts/buy-stamp.sh [amount] [depth]

# full wipe: down -v, rebuild bee + blockchain images --pull, up queen
./scripts/fresh.sh

# regenerate blockchain/state.anvil.json from upstream fdp-play-blockchain
UPSTREAM_TAG=2.2.0 ./scripts/rebake-blockchain.sh

# override pinned versions at build time
BEE_VERSION=2.8.0 docker compose build
FOUNDRY_VERSION=v1.5.1 docker compose build blockchain
```

APIs: queen `http://127.0.0.1:1633`, workers `http://127.0.0.1:{1,2,3,4}1633`. Chain RPC `:9545`.

## Architecture

**Three Dockerfiles, three roles, all baked.** Each service builds its own tagged image and copies its role-specific assets at build time:

- `bee/Dockerfile` — `FROM ethersphere/bee:${BEE_VERSION}`, takes a `ROLE` arg (`queen` | `worker-N`), `COPY`s `data/${ROLE}/` into `/home/bee/.bee/`. Each Bee service builds its own image (`bee-compose:queen-<ver>`, `bee-compose:worker-N-<ver>`).
- `blockchain/Dockerfile` — `FROM ghcr.io/foundry-rs/foundry:${FOUNDRY_VERSION}`, copies `blockchain/state.anvil.json` into `/state.anvil.json`. Entrypoint is `anvil --load-state /state.anvil.json` with chain-id 4020 and a 5s block time.
- Runtime Bee state lives in named volumes (`queen`, `worker-N`). The blockchain has **no volume** — the state is the image; tearing the container down resets the chain.

**The Anvil snapshot is the single source of truth for chain state.** `blockchain/state.anvil.json` is plain JSON (~720 KB) generated by `scripts/rebake-blockchain.sh` from `fairdatasociety/fdp-play-blockchain:${UPSTREAM_TAG}`. It contains:

- All 5 pinned Swarm contracts (factory, postage stamp, oracle, staking, redistribution) with their full bytecode and storage.
- ~10 helper/implementation contracts that the factory deploys at init.
- 11 genesis-funded EOAs (10000 ETH each) that the deterministic Bee swarm keys map to.
- 21 historical transactions + 19 event logs preserved through the dump/load cycle, so `eth_getLogs` works for events emitted during deployment.

The contract addresses pinned in `compose.yml`'s `x-bee-env` YAML anchor (factory, postage, oracle, staking, redistribution) match the snapshot. **Bumping the upstream tag means rebaking and re-checking those addresses haven't drifted** (they shouldn't, since the deployer EOA + nonce sequence is stable, but verify).

**Why an Anvil snapshot instead of upstream chaindata?** The upstream `fdp-play-blockchain` image bakes 141 MB of geth chaindata and runs in head-state-repair mode (`debug_dumpBlock` returns empty). It's also a heavy dependency for what's effectively a static state. Anvil + a small JSON snapshot gives us:

- ~720 KB instead of 141 MB.
- <1s startup instead of several seconds for geth init.
- No upstream image dependency at runtime — only at rebake time.

**Rebake mechanics (`scripts/rebake-blockchain.sh` + `rebake-replay.py`).** Conceptually simple, the implementation is forced into a roundabout path by limitations of the upstream chain:

- We can't use `debug_dumpBlock` / `debug_accountRange` — they return empty due to head-state repair.
- We can't use `debug_storageRangeAt` directly either — its keys are `keccak(slot)`, not original slots, and the upstream chaindata has no preimage cache. `anvil_setStorageAt` needs original slots.
- Solution: pull each tx's raw signed bytes via `eth_getRawTransactionByHash` and resubmit them via `eth_sendRawTransaction` to a scratch Anvil. Same chain-id (4020), same EIP-155 signatures → same contract addresses, same storage. Then post-correct balances + nonces (gas-rounding deltas are real but tiny) and `anvil_dumpState`.
- The dumped state is gzipped binary; `--load-state` wants plain JSON, so we gunzip before writing to disk.

**Role baked into the image, not mounted.** Re-emphasizing because it bites: `bee/Dockerfile` `COPY`s identity at build time. Editing `bee/data/` requires rebuilding the affected service's image (`docker compose build queen` or `... worker-N`).

**Pre-funded identities are paired with the snapshot.** `bee/data/{queen,worker-N}/keys/` holds deterministic libp2p / swarm / pss keys from `@fairdatasociety/fdp-play`. The Ethereum address Bee derives from each `swarm.key` matches one of the 11 genesis-funded EOAs. **Don't change the keys without rebaking** — the addresses are paired.

**Worker bootstrap is dynamic.** `scripts/workers-up.sh` curls `http://127.0.0.1:1633/addresses`, extracts the queen's peer id from `underlay`, and exports `QUEEN_BOOTNODE=/dns4/queen/tcp/1634/p2p/<peer-id>` before `docker compose --profile workers up -d`. DNS name (`queen`) is used instead of an IP so it survives container recreates; peer id is stable because it's derived from the baked libp2p key. The worker YAML anchor (`x-worker`) reads `${QUEEN_BOOTNODE:-}` — running `docker compose --profile workers up` without the script leaves workers with no bootnode.

**Workers are opt-in via profile.** `profiles: ["workers"]` on each worker service means plain `docker compose up` only starts `blockchain` + `queen`. Any command that should touch workers (including `down -v`) needs `--profile workers` — see `fresh.sh`.

## Gotchas

- **First-boot DNS race.** On a freshly created compose network, queen sometimes fails its first start with `dial tcp: lookup blockchain on ...: network is unreachable`. The `restart: unless-stopped` policy recovers within ~15s. If you're scripting against a clean stack and need determinism, `docker compose up -d --force-recreate` after bringing up the network avoids the race.
- **Stamp `not usable` on GET, but uploads work.** `GET /stamps/<id>` returns 400 "batch not usable" for ~30s after `buy-stamp.sh`, but `POST /bytes` with that stamp ID succeeds anyway. Bee's `IsUsable` check used by the GET endpoint is more conservative than the upload path. Same behavior holds against upstream fdp-play-blockchain.
- **Cross-peer retrieval may 404 without staking.** A chunk uploaded on the queen retrieves fine on the queen but may not retrieve via worker-N until kademlia topology stabilizes and stakes are placed. This is Swarm storage-incentives behavior, unrelated to the blockchain backend.
