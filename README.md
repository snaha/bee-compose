# bee-compose

A self-contained Docker Compose stack for running a local [Swarm](https://www.ethswarm.org/) Bee cluster against a dev blockchain. One queen + up to four workers, all pre-funded and ready to upload.

Useful for:

- developing apps against Bee without touching mainnet
- integration tests that need a real Bee + chain
- experimenting with multi-node behavior (replication, retrieval, neighborhood routing)

## What you get

| Service       | Container                  | Host port(s)         | Notes                              |
| ------------- | -------------------------- | -------------------- | ---------------------------------- |
| `blockchain`  | `bee-compose-blockchain`   | `9545` RPC           | Anvil booted from a baked snapshot |
| `queen`       | `bee-compose-queen`        | `1633` API, `1634` p2p | Full node, always running          |
| `worker-1..4` | `bee-compose-worker-N`     | `N1633`, `N1634`     | Light nodes, opt-in                |

The blockchain is **Anvil** (Foundry) loaded from `blockchain/state.anvil.json` — a state snapshot rebaked from `fairdatasociety/fdp-play-blockchain` that preserves every Swarm contract (factory, postage stamp, oracle, staking, redistribution) at its pinned address plus the 11 genesis-funded EOAs that Bee nodes draw gas from. Anvil starts in <1s and has no on-disk chaindata; the full state lives in the image layer.

Network ID `4020`. Contracts pinned in [`compose.yml`](./compose.yml) `x-bee-env`.

## Quick start

```bash
# queen + chain
docker compose up -d

# add the four workers (resolves queen's peer id, sets BEE_BOOTNODE)
./scripts/workers-up.sh

# buy a postage stamp on the queen (~24h headroom on a 5s-block chain)
./scripts/buy-stamp.sh

# tear everything down and rebuild from scratch (refreshes bee + foundry bases)
./scripts/fresh.sh
```

Queen API: <http://127.0.0.1:1633>. Workers: `http://127.0.0.1:{1,2,3,4}1633`.

## Configuration

- `BEE_VERSION` (default `2.7.1`) — pinned upstream Bee image tag. Override at build time:
  ```bash
  BEE_VERSION=2.8.0 docker compose build
  ```
- `FOUNDRY_VERSION` (default `stable`) — Foundry image tag for the Anvil blockchain.
- Worker count — edit `compose.yml`. Workers live behind the `workers` profile, so they only start when explicitly invoked (via `--profile workers` or `scripts/workers-up.sh`).
- Stamp parameters — `./scripts/buy-stamp.sh <amount> <depth>`. Defaults to `414720000` / depth `20`.

## How the pre-funding works

`bee/data/{queen,worker-N}/keys/` holds deterministic libp2p / swarm / pss keys originally generated by [`@fairdatasociety/fdp-play`](https://github.com/fairDataSociety/fdp-play). The Ethereum address Bee derives from each `swarm.key` matches an EOA pre-funded in `blockchain/state.anvil.json`. So on first boot:

1. Bee reads its baked keys.
2. Sees its account has gas, deploys its chequebook against the pre-deployed factory, and reaches `synced`.

Don't change the keys without rebaking the chain — the EOA addresses are paired.

## Rebaking the blockchain

The committed `blockchain/state.anvil.json` was produced by replaying every transaction from `fairdatasociety/fdp-play-blockchain:2.2.0` onto a scratch Anvil and dumping the resulting state. To regenerate (e.g. after bumping the upstream tag):

```bash
UPSTREAM_TAG=2.2.0 ./scripts/rebake-blockchain.sh
docker compose build blockchain   # bake the new state into the image
```

This is the only path that touches the upstream `fdp-play-blockchain` image. Day-to-day workflows (`up`, `down`, `fresh.sh`) don't need it.

## Layout

```
.
├── compose.yml                # services, env, ports, profiles
├── bee/
│   ├── Dockerfile             # FROM ethersphere/bee, COPYs role-specific keys
│   └── data/                  # pre-generated dev identities
├── blockchain/
│   ├── Dockerfile             # FROM foundry, COPYs the state snapshot
│   └── state.anvil.json       # baked Anvil state (committed)
└── scripts/
    ├── workers-up.sh          # resolves queen peer id, brings up workers profile
    ├── buy-stamp.sh           # POST /stamps with sane defaults, waits for settlement
    ├── fresh.sh               # nuke + rebuild + up
    ├── rebake-blockchain.sh   # regenerate state.anvil.json from upstream
    └── rebake-replay.py       # tx-replay helper invoked by rebake-blockchain.sh
```

## Prior art

- [`@fairdatasociety/fdp-play`](https://github.com/fairDataSociety/fdp-play) — the upstream "Bee + chain in a box" CLI. `bee-compose` is a leaner, compose-native take: no node CLI wrapper, no upstream geth image at runtime — just a `compose.yml`, an Anvil snapshot, and a handful of shell scripts. The dev identities and the upstream chain (used as the rebake source) come from fdp-play.
