# bee-compose

A self-contained Docker Compose stack for running a local [Swarm](https://www.ethswarm.org/) Bee cluster against a dev blockchain. One queen + up to four workers, all pre-funded and ready to upload.

Useful for:

- developing apps against Bee without touching mainnet
- integration tests that need a real Bee + chain
- experimenting with multi-node behavior (replication, retrieval, neighborhood routing)

## What you get

| Service       | Container                  | Host port(s)         | Notes                              |
| ------------- | -------------------------- | -------------------- | ---------------------------------- |
| `blockchain`  | `bee-compose-blockchain`   | `9545` RPC, `9546` WS | `fdp-play-blockchain` dev geth     |
| `queen`       | `bee-compose-queen`        | `1633` API, `1634` p2p | Full node, always running          |
| `worker-1..4` | `bee-compose-worker-N`     | `N1633`, `N1634`     | Light nodes, opt-in                |

The blockchain image is `fairdatasociety/fdp-play-blockchain` — its genesis pre-funds the chequebook addresses derived from the keys baked into `bee/data/{queen,worker-N}/keys/`. Bee skips its normal funding step and boots straight into a working state.

Network ID `4020`. Contracts (factory, postage stamp, oracle, staking, redistribution) are pinned to the addresses produced by the genesis — see [`compose.yml`](./compose.yml) `x-bee-env`.

## Quick start

```bash
# queen + chain
docker compose up -d

# add the four workers (resolves queen's peer id, sets BEE_BOOTNODE)
./scripts/workers-up.sh

# buy a postage stamp on the queen (~24h headroom on a 5s-block chain)
./scripts/buy-stamp.sh

# tear everything down and rebuild from scratch (pulls latest bee + blockchain)
./scripts/fresh.sh
```

Queen API: <http://127.0.0.1:1633>. Workers: `http://127.0.0.1:{1,2,3,4}1633`.

## Configuration

- `BEE_VERSION` (default `2.7.1`) — pinned upstream Bee image tag. Override at build time:
  ```bash
  BEE_VERSION=2.8.0 docker compose build
  ```
- Worker count — edit `compose.yml`. Workers live behind the `workers` profile, so they only start when explicitly invoked (via `--profile workers` or `scripts/workers-up.sh`).
- Stamp parameters — `./scripts/buy-stamp.sh <amount> <depth>`. Defaults to `414720000` / depth `20`.

## How the pre-funding works

`bee/data/{queen,worker-N}/keys/` holds deterministic libp2p / swarm / pss keys originally generated by [`@fairdatasociety/fdp-play`](https://github.com/fairDataSociety/fdp-play). The chequebook address Bee derives from each `swarm.key` matches an account that the `fdp-play-blockchain` genesis funds. So on first boot:

1. Bee reads its baked keys.
2. Sees its chequebook is already deployed and funded.
3. Skips the funding/deploy dance and goes straight to `synced`.

Don't change the keys unless you also change the genesis.

## Layout

```
.
├── compose.yml          # services, env, ports, profiles
├── bee/
│   ├── Dockerfile       # FROM ethersphere/bee, COPYs role-specific keys
│   └── data/            # pre-generated dev identities (gitignored runtime dirs)
└── scripts/
    ├── workers-up.sh    # resolves queen peer id, brings up workers profile
    ├── buy-stamp.sh     # POST /stamps with sane defaults, waits for settlement
    └── fresh.sh         # nuke + rebuild + up
```

## Prior art

- [`@fairdatasociety/fdp-play`](https://github.com/fairDataSociety/fdp-play) — the upstream "Bee + chain in a box" CLI. `bee-compose` is a leaner, compose-native take: no node CLI wrapper, just a `compose.yml` and three shell scripts. The dev identities and blockchain image come from fdp-play.
