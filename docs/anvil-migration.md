# Anvil migration

This is the decision record for swapping the blockchain backend from
`fairdatasociety/fdp-play-blockchain` (geth) to Anvil booted from a baked
state snapshot. Written so that someone reading this repo cold understands
why the `blockchain/` directory looks the way it does and can confidently
bump the upstream tag without re-deriving the whole approach.

## Context

The upstream image bundles three concerns:

1. A geth dev chain (network ID 4020, 5s blocks, single unlocked etherbase).
2. ~141 MB of pre-populated chaindata containing 21 transactions in block 5
   that deploy 5 pinned Swarm contracts (factory, postage stamp, price
   oracle, staking, redistribution) plus ~10 helper/implementation
   contracts they create internally.
3. A genesis with 11 EOAs pre-funded at 10 000 ETH each. The Bee swarm
   keys baked into `bee/data/{queen,worker-N}/keys/` derive Ethereum
   addresses that are subsets of those 11 EOAs — so on first boot Bee has
   gas without any faucet step.

The image works, but it's a heavy runtime dependency:

- 141 MB chaindata in the image layer.
- Multi-second startup while geth opens the database and runs head-state
  repair (`Head state missing, repairing` at every boot — see below).
- Bound to a single upstream tag we can't independently version.
- Black-box: introspection via `debug_dumpBlock` returns empty, so even
  understanding "what's in here?" requires bespoke probing.

## Options considered

| Option | What it looks like | Pros | Cons |
| --- | --- | --- | --- |
| **A. Freeze the chaindata** | `docker cp` the 141 MB `geth/chaindata` out of the upstream once, commit it, build our own image FROM `ethereum/client-go`. | No new tooling. Contract addresses preserved trivially. | Still 141 MB in the image. Inherits head-state-repair behaviour. Snapshot is opaque (binary LDB). |
| **B. Deploy from source** | Pull `ethersphere/storage-incentives` at a pinned tag, hardhat/foundry deploy script, produce genesis from scratch. | Full transparency. Can bump contract versions independently of fdp-play. | New project under `blockchain/deploy/` — Solidity pin, deploy scripts, Bee bootstrap dependency on the deploy completing. Significant new surface. |
| **C. Anvil + state snapshot** | Rebake state from upstream once via tx replay, commit a small JSON snapshot, runtime image is just `foundry + state.anvil.json`. | <1 MB snapshot. <1s startup. Snapshot is human-inspectable JSON. No upstream image at runtime. | One-time complexity in the rebake pipeline. Future contract version bumps still require fdp-play to update first (unless paired with B). |

We picked **C**. **B** is the medium-term direction once we want to track
contract upgrades independently — see "Future work" below.

## Decision

`compose.yml` runs `ghcr.io/foundry-rs/foundry:stable` as the blockchain
service. The container's only state is a baked-in `state.anvil.json` file
loaded via `anvil --load-state`. The state file is checked into the repo
under `blockchain/state.anvil.json` and was generated by
`scripts/rebake-blockchain.sh` from
`fairdatasociety/fdp-play-blockchain:2.2.0`. Bumping that upstream tag is
done by running the rebake script and committing the new snapshot —
nothing else in the repo needs to change.

Contract addresses pinned in `compose.yml`'s `x-bee-env` are unchanged
from before: factory `0xCfEB…0a601`, postage stamp `0x254d…ABC2B`, price
oracle `0x5b18…526b24`, staking `0xD833…f841bb`, redistribution
`0x9561…0f06a7`. They survive rebakes because the rebake re-executes the
same signed transactions from the same deployer EOA at the same nonces.

## Why "tx replay" — and not state copy

The instinctive approach is "extract every account's balance / nonce /
code / storage from upstream, replay via Anvil's `anvil_setBalance` /
`anvil_setCode` / `anvil_setStorageAt`, dump." That's what the script
tried first. Three things stop it:

1. **`debug_dumpBlock` returns `{accounts: {}}` on this chain.** geth
   logs `Head state missing, repairing` on boot and falls back to a
   point-lookup mode where individual `eth_getCode` and
   `debug_storageRangeAt` queries work but no full trie iteration does.
   So we can't enumerate accounts via the standard dump endpoint.

2. **`debug_storageRangeAt` returns hashed slot keys, not original.**
   Geth iterates the storage trie keyed by `keccak(slot)` and only
   surfaces the original slot in the response if a preimage cache is
   populated. The upstream image's chaindata has no preimages, so every
   `key` field comes back `null`. `anvil_setStorageAt` needs original
   slots, so we can't replay storage even when we can read it.

3. **Mapping-based storage is unenumerable from the outside.** Even with
   working trie iteration, slot layouts of compiled Solidity (mappings
   live at `keccak(key . slot)`) mean you can't reconstruct the keys from
   storage values alone.

The escape hatch: **`eth_getRawTransactionByHash` works.** It returns the
fully signed RLP bytes of any historical transaction. Submitting those
bytes to a fresh Anvil with the same chain-id (4020) re-executes them
under EIP-155 — same signer, same nonces, same CREATE addresses, same
constructor arguments, same resulting storage. The whole "what's in
storage" question becomes "what did the constructors and post-deploy
calls write?" — answered by the EVM itself, not by us.

Concretely the rebake does:

1. Boot upstream geth (`fairdatasociety/fdp-play-blockchain:${UPSTREAM_TAG}`)
   and wait for it to be responsive.
2. Read its `genesis.json` from `/root/genesis.json` (this gives us the
   pre-funded EOA balances; the genesis itself contains no contracts).
3. Boot a scratch Anvil with `--chain-id 4020` and automine.
4. Apply every genesis allocation via `anvil_setBalance`.
5. Walk every block on upstream, fetch each tx's raw bytes with
   `eth_getRawTransactionByHash`, submit them to Anvil with
   `eth_sendRawTransaction`. Every receipt is checked for
   `status == "0x1"` — any failure aborts the rebake.
6. Walk the same blocks again to enumerate every address that appeared
   (via `from`, `to`, `contractAddress`, and `debug_traceTransaction`
   call-tracer for internal CREATEs). Add the genesis EOA addresses.
7. For each enumerated address, copy `eth_getBalance` and
   `eth_getTransactionCount` from upstream onto Anvil. This corrects for
   gas-rounding differences (Anvil's gas pricing differs slightly from
   geth's, so the deployer EOA loses a few wei more than upstream after
   replaying the same txs).
8. Call `anvil_dumpState` — this returns hex of gzipped JSON.
9. Decompress and write to `blockchain/state.anvil.json`. We store
   plain JSON because `anvil --load-state` rejects gzipped files
   ("stream did not contain valid UTF-8") and the file is small enough
   (~720 KB) that compression isn't worth the runtime decompression
   step.

`scripts/rebake-blockchain.sh` orchestrates the Docker side;
`scripts/rebake-replay.py` does the JSON-RPC choreography.

## What's preserved in `state.anvil.json`

After rebake, the snapshot contains:

- All 5 pinned Swarm contracts at their pinned addresses, plus ~10
  helper/implementation contracts (most of them deployed internally by
  the factory's constructor).
- The 11 genesis-funded EOAs with 10 000 ETH each (modulo the deployer's
  ~5 ETH gas spend on block 5).
- 21 transaction records from block 5 and the resulting block headers.
- 19 event logs emitted during deployment, queryable via `eth_getLogs`
  on the loaded chain.

What's *not* preserved: anything created by Bee on first boot against the
old image — chequebook contracts, postage batches, kademlia state. Bee
recreates those from scratch on its first run against the snapshot, same
as it would against a fresh upstream geth. The snapshot is "the chain
state right after the contracts were deployed but before any Bee node
talked to it" — exactly the state someone needs to start from zero.

## Verification done at rebake time

After applying the snapshot and before declaring victory, this was
checked:

| Property | Method | Result |
| --- | --- | --- |
| Contract bytecode (8 contracts) | `eth_getCode` byte-for-byte vs upstream geth | identical |
| Storage spot-checks | `eth_getStorageAt` on slots seen in `debug_storageRangeAt` | identical |
| Deployer nonce | `eth_getTransactionCount` | 21 on both |
| Event logs | `eth_getLogs(fromBlock=0, toBlock=latest)` | 19 logs on both |
| Balance parity | All enumerated addresses post-corrected to upstream values | identical |

End-to-end, against a freshly built `bee-compose` stack:

- Anvil boots in <1s.
- Queen reaches `/health: ok` and `/readiness: ready`.
- Queen deploys its chequebook on first boot (the deploy goes through
  the rebaked factory, against the rebaked storage layout).
- All 4 workers boot, deploy chequebooks, and connect to queen as
  libp2p peers (`/peers` returns 4).
- `scripts/buy-stamp.sh` settles a postage batch within ~30s and
  uploads using that batch ID succeed.

## Gotchas worth flagging

- **First-boot DNS race.** On a brand-new compose network, queen
  occasionally fails its first start with
  `dial tcp: lookup blockchain on …: network is unreachable`.
  `restart: unless-stopped` recovers it within ~15s. Force-recreating
  (`docker compose up -d --force-recreate`) avoids the race when
  scripting against a clean stack.

- **`GET /stamps/<id>` returns 400 "batch not usable" for ~30s after
  purchase, but `POST /bytes` with that same batch ID succeeds
  immediately.** Bee's `IsUsable` check used by the GET endpoint is
  stricter than the upload path. This held against upstream
  fdp-play-blockchain too — it's a Bee API quirk, not specific to Anvil.

- **Cross-peer retrieval may 404 without staking.** A chunk uploaded on
  the queen retrieves fine via the queen but may not retrieve via
  worker-N until kademlia topology stabilizes and stakes are placed.
  This is Swarm storage-incentives behaviour and is independent of the
  blockchain backend.

- **Block numbering shifts between snapshots.** Upstream geth has the
  21 deploy txs in a single block 5; Anvil with automine puts them in
  21 separate blocks. The rebaked snapshot therefore starts at block 21
  (Anvil's `best_block_number` after replay). Bee scans from block 1
  for postage stamp events, so this doesn't matter operationally — but
  if anything in your tooling pins to "the contracts are at block 5",
  it won't be after the rebake.

## Future work

- **Deploy from source** (Option B above). When we want to track Bee
  upstream contract version bumps independently of fdp-play, replace
  `scripts/rebake-blockchain.sh`'s "fetch txs from upstream" step with
  a foundry/hardhat deploy against `ethersphere/storage-incentives`
  pinned at a tag. The downstream of the rebake (apply genesis allocs +
  replay → dump state) stays the same; the upstream becomes our own
  deploy pipeline.

- **Activate the price oracle in the snapshot.** The `GET /stamps`
  cosmetic 400 hangs on the price oracle being inactive (oracle slot 0
  set, but price = 0). A post-deploy `setPrice(...)` call in the rebake
  would clean this up. Low priority — uploads work either way.

- **CHANGELOG.** Once this repo gets published and starts taking version
  tags, this file becomes the entry under "0.x → 1.0" or wherever the
  Anvil swap lands.
